<fix_guidance>
The root cause: `express.json()` is registered as global middleware ahead of the `/webhook` route, so a JSON parse failure short-circuits to Express's default error handler before the route handler (where the signature check lives) ever runs. The fix must make signature verification happen against the raw body bytes BEFORE any JSON-parsing failure can short-circuit the request — for every code path, not just the ones where JSON.parse happens to succeed.

A standard, idiomatic fix: stop relying on `express.json()`'s `verify` callback (which fires ON the way to parsing, but a parse failure inside the same middleware still short-circuits past the route handler). Instead, read the raw body for the `/webhook` route with `express.raw({ type: 'application/json' })` (an Express/body-parser built-in, no new dependency) so the route handler receives a `Buffer` unconditionally, verify the HMAC signature over that raw buffer first, and only THEN attempt `JSON.parse` on it inside the route handler (a syntax error there is a body-shape problem → `400 { error: 'invalid_body' }`, not a signature problem, and it happens strictly after the signature check passes). Keep `express.json()` for the other existing routes if any of them rely on parsed JSON (currently none do — `GET /health`, `GET /items`, `GET /items/:id` take no body), or scope `express.json()`/`express.raw()` per-route so `/webhook` alone uses the raw-body reader.

Do not hand-roll a body-parsing module — `express.raw()` is the standard built-in primitive for this exact "verify raw bytes before parsing" pattern; using it is not a new dependency (it ships inside `express`/`body-parser`, already a transitive dependency).
</fix_guidance>
