## Risks

- **Out-of-scope refusals**: do not touch `server/`, `web/`, `data/payout-rules.json`, `package.json`, or any file outside the authorized surface. Do not add a new `dependencies`/`devDependencies` entry (Constraint: no new deps — implementation must use only `fs`, `path`, and core Node, matching `bin/cli.js:5-6`'s existing requires). Do not add flags/commands beyond `payout --input <path>`; `hello`, `version`, `--help` stay byte-for-byte behaviorally unchanged (`bin/cli.js:38-61`).
- **No hardcoded rule values**: `processing_fee_percent`, `fixed_fee_cents`, `dispute_fee_cents`, `reserve_percent`, `minimum_payout_cents` must be read from `data/payout-rules.json` (currently `2.9`, `30`, `1500`, `10`, `1000` — see `data/payout-rules.json:1-7`) via `fs.readFileSync` + `JSON.parse` at command run time, mirroring the existing `readPackageVersion()` pattern at `bin/cli.js:21-25`, not copied as literals into the new payout logic.
- **No silent catch**: file-read and `JSON.parse` failures for both the input file and (implicitly) `data/payout-rules.json` must be caught and explicitly turned into a printed JSON error + `process.exit(2)`, never a bare `try {} catch {}` that swallows and falls back. This applies to the *input* file per Requirement 4 ("unreadable input file", "invalid JSON in the input file"); `data/payout-rules.json` is repo-internal and not itself part of the validation-failure contract, but its read must still not be silently swallowed.
- **Exact literal to match**: the conflicting-duplicate error is spec'd verbatim — stdout must be empty, and stderr must be exactly `{"error": "conflicting_duplicate", "id": string}` serialized with no extra whitespace, i.e. `JSON.stringify({ error: 'conflicting_duplicate', id })` which produces `{"error":"conflicting_duplicate","id":"<id>"}` (matches the Verification block's exact string, Requirement 2 / Verification line 3). Other validation-failure error objects are not given a literal shape in the spec beyond "exactly one JSON error object" on stderr with nothing on stdout — implementation must pick a single consistent `{"error": "<code>", ...}` shape and apply it uniformly across all validation-failure branches (unknown `type`, missing `merchant_id`, missing `id`, non-positive/non-integer `amount_cents`, missing `events` array, invalid JSON, unreadable file); do not invent per-branch ad hoc shapes.
- **"Byte-identical JSON content" ambiguity — interpret strictly but practically**: the spec's literal wording ("byte-identical") is not achievable after `JSON.parse` (which discards source formatting/whitespace/key order). Interpretation adopted: two events sharing an `id` are treated as identical duplicates when their parsed fields (`id`, `merchant_id`, `type`, `amount_cents`) are all deep-equal; any field difference is a conflicting duplicate. This is the only interpretation consistent with "seen again with byte-identical JSON content is applied only once" being checkable post-parse, and must not be second-guessed into e.g. a raw-substring comparison (there is no accessible raw substring per-event once the whole file is `JSON.parse`d).
- **Validation-vs-dedup ordering**: Requirement 3 lists validation failures (unknown type, missing `merchant_id`/`id`, bad `amount_cents`, missing `events` array, invalid JSON, unreadable file) as failing "before any totals are computed or printed" — same as the conflicting-duplicate case. Interpretation adopted: process events in original array order; for each event, validate its shape first (an event missing `id` can't be deduped, so shape validation must precede the dedup lookup), then perform the dedup/conflict check, then apply it to per-merchant accounting only if it's a new id. The first invalid or conflicting event encountered (in original order) triggers immediate exit 2 with nothing further computed — do not collect/report multiple errors.
- **Merchant row ordering**: "merchant rows are ordered by first-seen merchant" after duplicate removal — use an insertion-ordered `Map` keyed by `merchant_id`, not a sort, so ties resolve to first-appearance order in the deduped event stream.
- **Reserve/threshold rule is a single conditional, not two independent adjustments**: `reserve_cents = Math.round(net_before_reserve * reserve_percent / 100)` only when `net_before_reserve > 0` (else `0`), then `payout_cents = net_before_reserve - reserve_cents`; only *after* that, if `0 < payout_cents < minimum_payout_cents`, roll the original positive `payout_cents` into `reserve_cents` and zero `payout_cents` (Requirement 3, last two sentences). Do not apply the minimum-threshold rollback when `payout_cents <= 0` already.
- **`--input` flag parsing**: the spec does not enumerate "missing `--input` flag/value" as one of the JSON-error validation failures (Requirement 4's list is about event/file *content*, not CLI argument shape). Interpretation adopted, consistent with the existing `--name` convention at `bin/cli.js:27-36`: a missing `--input` flag or missing value is a plain-text usage error to stderr with `process.exit(1)` (like `parseNameFlag`), not a JSON error object — this is an interpretation of an unspecified case, not a literal requirement, and is not exercised by the Verification block.
- **Test-harness detail**: `execFileSync` (used by every existing test, `tests/cli.test.js:8-10`) throws on non-zero exit status, so the new validation-failure test must wrap the call in `try/catch` and assert on the caught error's `status`, `stdout`, and `stderr` properties (`error.status === 2`, `error.stdout` is empty, `error.stderr` parses as the expected JSON) rather than calling the existing `run()` helper directly and expecting a normal return.
- **Speculative-robustness refusal**: no extra flags (`--output`, `--format`, etc.), no persistence of results, no new npm scripts, no README/doc updates — none are requested and Out of Scope explicitly forbids anything beyond the `payout --input <path>` command itself.
