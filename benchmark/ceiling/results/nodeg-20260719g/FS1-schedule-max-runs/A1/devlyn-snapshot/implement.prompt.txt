## Task

Implement `Job.max_runs(n)` exactly per `.devlyn/plan.md` (read it first) and `.devlyn/criteria.generated.md`. Key points already decided by PLAN (do not re-litigate, just execute):

- `Job.max_runs(n)` is chainable (returns `self`), usable anywhere in the fluent chain.
- Validation: `n` must satisfy `isinstance(n, int) and n >= 1`; otherwise raise `schedule.ScheduleValueError` (this codebase's established convention for job-configuration validation errors — see `Job.until()` / `Job.at()` in `schedule/__init__.py`), at `.max_runs(...)` call time.
- The budget counts actual executions of the job's function (decrement strictly after `job_func()` runs inside `Job.run()`, never before — an execution that's skipped due to `.until()` overdue must not consume budget).
- After the nth execution, the job's `run()` must return the `CancelJob` sentinel so `Scheduler._run_job()`'s existing removal path (`isinstance(ret, CancelJob)` -> `cancel_job`) removes it — do not add a second removal mechanism. This covers `run_pending()` and `run_all()` automatically since both funnel through `_run_job()`.
- First-limit-wins with `.until(deadline)`: whichever of (budget exhausted, deadline passed) is hit first cancels the job; do not touch the existing `_is_overdue()` / `cancel_after` logic, add the budget check as an independent, additional condition.
- Calling `.max_runs()` more than once: last call wins (unconditionally overwrite the remaining-count state).
- Per-job: plain instance attribute, no shared/class state.
- No public getter/property for the remaining budget — out of scope.
- Document: a proper docstring on `Job.max_runs()` (picked up by Sphinx autodoc in `docs/reference.rst` — do NOT edit `docs/reference.rst` itself, it already autodocuments `Job` with `:members:`) plus a new prose + code-block example section in `docs/examples.rst`, placed after the existing "Run a job until a certain time" section, mirroring its style.
- Tests: add a test method (e.g. `test_max_runs`) in `test_schedule.py`'s `SchedulerTests`, near `test_until_time`, using the existing `make_mock_job()` / `mock_datetime()` helpers. Cover: chainability at either chain position, exact-n-then-cancel via both `run_pending()` and `run_all()`, first-limit-wins vs `.until()`, validation errors (non-int, `0`, negative, `None`, float) raising `ScheduleValueError`, last-call-wins, and no cross-job interference (a second concurrently scheduled job with a different/no budget is unaffected). Do not modify `test_until_time` or any other existing test.
