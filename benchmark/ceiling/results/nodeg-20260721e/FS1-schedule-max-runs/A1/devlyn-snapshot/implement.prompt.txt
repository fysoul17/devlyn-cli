The task is:

Read `.devlyn/plan.md` (the approved plan) and `.devlyn/criteria.generated.md` (the full spec) in the repo root, then implement exactly what they specify. Summary: add a run-budget feature to the `schedule` Python library (this repository, `schedule/__init__.py`).

Implement `Job.max_runs(n)`:

- Chainable setter that returns the Job instance, usable anywhere in the fluent chain, e.g. `schedule.every(5).seconds.max_runs(3).do(job_fn)` or `schedule.every().max_runs(3).seconds.do(job_fn)`.
- The budget counts actual executions of the job function. After the job has executed n times in total, the job removes itself from its scheduler exactly as a job returning `CancelJob` does. The nth execution itself still runs normally; an (n+1)-th execution must never happen.
- Applies to every execution path that runs jobs through the scheduler (`run_pending()` and `run_all()` included).
- Interaction with `.until(deadline)`: whichever limit is reached first cancels the job (first-limit-wins). The existing `.until()` semantics must not regress.
- After a job exhausts its budget, `Scheduler.next_run` / `idle_seconds` must reflect its removal.
- Validation: n must be an integer >= 1; any other value raises builtin `ValueError` (not `ScheduleValueError` — this is a deliberate, literal divergence from this codebase's usual setter-validation convention; do not "fix" it) at `.max_runs(...)` call time. Calling `.max_runs()` more than once on the same job: the last call wins.
- Budgets are per-job: multiple jobs with different budgets must not interfere with each other.
- Document the new API with a docstring consistent with sibling methods (`tag()`, `to()`, `until()`) — `docs/reference.rst` autodocs `schedule.Job` with `:members:`, so no separate doc-page edit is needed or in scope.

The approved plan (`.devlyn/plan.md`) directs:
- Storage attributes must NOT be named `max_runs` (would shadow the bound method after the first call and break a second `.max_runs()` call). Use distinct public attribute names for the budget and the execution counter, consistent with the class's existing convention (`tag()`→`self.tags`, `to()`→`self.latest`, `until()`→`self.cancel_after`).
- Integrate via a new predicate mirroring `Job._is_overdue()`, OR'd into the single existing post-run cancellation check inside `Job.run()` — reuse `Scheduler._run_job`'s existing `CancelJob` handling; add no scheduler-side bookkeeping.
- Only a post-run check is needed (no pre-run check) — a budget-exhausted job is already removed from `scheduler.jobs`, so there's no reachable "already exhausted" pre-run state.
- Out of scope: `docs/examples.rst`, `docs/reference.rst`, `HISTORY.rst`/`docs/changelog.rst`, `README.rst`, and any `repeat()` behavior change.

Deliver production-quality code: the full existing test suite must keep passing, and the new behavior must be covered by tests you add in `test_schedule.py` (existing file uses plain `unittest.TestCase` style; it already defines a `mock_datetime` context-manager helper — reuse it for the `.until()`-interaction test; use `run_all()` for pure execution-count tests the way the existing `test_cancel_job` test does, per the plan).
