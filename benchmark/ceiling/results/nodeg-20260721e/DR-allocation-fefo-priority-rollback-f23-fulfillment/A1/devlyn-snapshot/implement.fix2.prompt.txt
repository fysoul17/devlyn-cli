## Root cause

`{ warehouse: warehouse.id, ...lot }` sets `warehouse` first, then spreads `lot`'s own keys on top — if the input `lot` object happens to carry its own `warehouse` key (the Lot schema does not forbid unknown/extra keys), the spread silently overwrites the correct `warehouse.id` value. This is the classic JS object-spread field-collision footgun.

## Required fix

Remove the spread. Build the flattened row with explicit fields only: `warehouse: warehouse.id, sku: lot.sku, lot: lot.lot, qty: lot.qty, expires: lot.expires` (plus whatever internal field you still need for the qty>0 filter and comparator — check `lotComparator`'s exact usage, since it currently expects `.expires`/`.lot` on its arguments, so keep that compatible). Since the row will now already carry exactly the final `{warehouse, sku, lot, qty, expires}` shape with no risk of extra/spoofed keys, the trailing `.map(({ warehouse, sku, lot, qty, expires }) => ({ warehouse, sku, lot, qty, expires }))` whitelist step (added in round 1 specifically to strip spread pollution) becomes redundant — remove it too if the earlier explicit construction already produces the exact clean shape (net simplification, not scope creep — this directly follows from removing the root cause of the finding you're fixing).
