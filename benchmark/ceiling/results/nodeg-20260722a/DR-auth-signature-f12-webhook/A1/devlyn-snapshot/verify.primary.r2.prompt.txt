# PHASE 5 — VERIFY / JUDGE (primary, broad-coverage, round 2) — fresh worker, findings-only

**You are spawned with empty conversation context.** You have no code-mutation tools — findings only. Do not edit any file. Do not run `apply_patch`.

<role>
Independent quality layer. Answer one question: did the diff deliver what the spec said it would, with no scope creep, no quality regression, no constraint violation?
</role>

<input>
- Spec: `.devlyn/criteria.generated.md` in this repo (generated free-form criteria — Requirements / Constraints / Out of Scope / Assumptions / Verification).
- The cumulative diff: `git diff d5e479312b6f9573373bd2057e630bba7d22c608..HEAD -- server/index.js tests/server.test.js`.
- You do NOT receive PLAN's or IMPLEMENT's reasoning, or BUILD_GATE's findings, or a prior VERIFY round's findings — read the spec and diff fresh and form your own judgment.
</input>

<task>
Grade the diff against the spec on these axes:

- **Spec compliance** — does every Requirement in `.devlyn/criteria.generated.md` have code that satisfies it? Quote file:line evidence.
- **Scope** — does the diff touch only `server/index.js` and `tests/server.test.js`? Any other file changed = HIGH finding `scope.out-of-scope-violation`.
- **Quality** — idiomatic Express/Node patterns, or hand-rolled logic replacing a standard primitive? MEDIUM `design.unidiomatic-pattern` if so.
- **Consistency** — style consistent with the surrounding code (naming, error-response shape, module boundaries).

**Clause-level check** — split each Requirement into binding clauses. Trace the actual code order in `server/index.js`:
1. Does the signature check genuinely run *before* body-shape validation, for ALL bodies including malformed/non-JSON/primitive bodies (not just well-formed-but-wrong-shape JSON objects)?
2. Does duplicate-id detection genuinely happen *after* both signature and shape checks, and does it ignore the second body's `type`/`timestamp`/`data`?
3. Is `crypto.timingSafeEqual` guarded against a length mismatch (must not throw)?
4. Is the raw body actually the exact bytes signed (not a re-serialized/re-parsed value)?
5. Does a missing/malformed `X-Signature` header fold into the same 401 path rather than throwing an uncaught exception or leaking a generic framework error page?
6. For ANY body that is not valid JSON, or is valid JSON but not an object (e.g. a bare number, string, array, or `null`), does the route return the EXACT documented error shapes (`401 {"error":"invalid_signature"}` when the signature is wrong/missing, or `400 {"error":"invalid_body"}` when the signature is valid but the shape fails) — never a generic Express/framework error page (HTML, stack trace, or any response whose body isn't exactly the documented JSON shape)?

**Interaction check** — construct at least one adversarial scenario combining two verification bullets and actually run it against the live code (start the server via a temporary throwaway script; leave no tracked files behind).

**Coverage check** — if you cannot exercise an axis, say so explicitly rather than assuming a pass.

**Severity** — HIGH/CRITICAL findings are always verdict-binding. A MEDIUM finding is verdict-binding only if it's a concrete behavioral regression against the spec or existing test contract; otherwise it's advisory (PASS_WITH_ISSUES).

**Anti-self-filter** — report every finding you observe, including low-severity/low-confidence ones. Tag `confidence: high|medium|low`.
</task>

<output>
Emit JSONL findings, one per line, to stdout — each line: `{"id": "...", "severity": "CRITICAL|HIGH|MEDIUM|LOW", "rule_id": "...", "file": "...", "line": N, "message": "...", "confidence": "high|medium|low"}`. If zero findings, print exactly one line: `{"verdict": "PASS"}`. End your reply with a one-line summary verdict: `PASS`, `PASS_WITH_ISSUES`, or `NEEDS_WORK`.
</output>
