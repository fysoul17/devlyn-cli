<findings>
1. **HIGH — `spec.signature-check-order`** (`server/index.js`): The global `app.use(express.json({ verify: ... }))` middleware parses the request body for ALL routes, including `/webhook`, before the route handler runs. When the request body is not valid JSON (e.g. malformed/garbled bytes), `express.json()`'s strict parser throws a `SyntaxError`, which Express's default error handler catches and returns as `400` with an HTML body — this happens BEFORE the `/webhook` route handler code (which contains the signature check) ever executes. Per the spec, the signature check must run first for every request (it only needs the raw bytes, not parsed JSON), so a malformed-JSON request with a missing/invalid signature must still get `401 {"error":"invalid_signature"}`, not a generic Express `400` HTML error. Reproduced: `POST /webhook` with body `not-json-at-all{{{` and no `X-Signature` header returns `400 text/html` (Express's default parse-error page), not `401 {"error":"invalid_signature"}`.

2. **HIGH — `spec.invalid-body-response`** (`server/index.js`): `express.json()` defaults to `strict: true`, which rejects any top-level JSON value that isn't an object or array (e.g. a bare JSON primitive like `42`) with its own `SyntaxError` → Express's default `400` HTML error page, again bypassing the route handler entirely. A body that "fails the shape" (per spec, `{ id, type, timestamp, data }` required) must return exactly `400 {"error":"invalid_body"}` — not a generic HTML error. Reproduced: a correctly-signed body of `42` returns `400 text/html`, not `400 {"error":"invalid_body"}`.

**Root cause (both findings)**: the `/webhook` route currently relies on the *global* `express.json()` middleware to both capture the raw body (via its `verify` hook) and parse it — but that middleware's own strict-mode JSON parsing can fail and short-circuit to Express's default error handler before the `/webhook` handler's signature-check-first logic ever runs. The webhook route needs to control its own raw-body acquisition, independent of whether the bytes are valid/strict JSON, so that: (a) the signature check always runs first against the raw bytes regardless of JSON validity, and (b) any JSON-parse failure or shape failure is handled inside the route as `400 {"error":"invalid_body"}`, never as Express's generic parser error page.
</findings>
