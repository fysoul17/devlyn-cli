Both judges independently and reproducibly confirmed JUDGE-1/PAIR-1's core claim (same root cause): `Job.run()` only checks the run-budget AFTER calling `job_func()`, never BEFORE. The existing `.until()` deadline guard, by contrast, has a pre-run check (`if self._is_overdue(datetime.datetime.now()): return CancelJob` at the top of `run()`, before `job_func()` is called) — that is precisely what makes `.until()` safe against a second call after the deadline, whether reached via normal scheduling or via any other path that re-invokes `run()`. `max_runs()` needs the analogous pre-run guard: `Job.run()` must check `if self._max_runs is not None and self._run_count >= self._max_runs` and return `CancelJob` WITHOUT calling `job_func()` when the budget is already exhausted at the top of `run()`, in addition to (not instead of) the existing post-run check that catches the exact-nth-run case. This closes the gap for BOTH the "already exhausted, called again directly" case AND the reachable-via-public-API case where `.max_runs()` is reconfigured downward to at/below the already-accumulated count (Requirement 7) and the scheduler runs the job again before the caller notices — the fix must not let that scheduler run call `job_func` at all once the budget is already met or exceeded.
