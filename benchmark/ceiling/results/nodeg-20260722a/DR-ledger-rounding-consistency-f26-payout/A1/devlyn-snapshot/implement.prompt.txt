The task: implement `bench-cli payout --input <path>` (i.e. `node bin/cli.js payout --input <path>`) and its tests, exactly per `.devlyn/criteria.generated.md` and `.devlyn/plan.md`. Both files are in this repo at those paths — read them in full before writing any code. Key points already resolved by PLAN (do not re-litigate):

- Idempotency: same `id` + deep-equal (parsed, key-order-insensitive) event content ⇒ applied once. Same `id` + different content ⇒ `conflicting_duplicate`: exit 2, stderr exactly `{"error":"conflicting_duplicate","id":"<id>"}`, no stdout.
- Field validation (unknown `type`, missing `merchant_id`, missing `id`, non-integer/non-positive `amount_cents`, missing `events`, invalid JSON, unreadable file) runs in array order; the first problem encountered (field validation or conflicting duplicate) is reported; field validation for an event runs before its dedup check. Every such failure: exit 2, exactly one JSON error object to stderr, nothing to stdout.
- Per-merchant accounting, in first-seen-after-dedup order: `charge` → `gross_charge_cents += amount_cents`; `processing_fee_cents += Math.round(amount_cents * processing_fee_percent / 100) + fixed_fee_cents` (computed per charge event, then accumulated — not rounded once on the aggregate). `refund` → `refund_cents += amount_cents`. `dispute` → `dispute_cents += amount_cents`; `dispute_fee_cents += <configured dispute_fee_cents>`.
- Then per merchant: `net_before_reserve = gross_charge_cents - refund_cents - dispute_cents - processing_fee_cents - dispute_fee_cents`; `reserve_cents = Math.round(net_before_reserve * reserve_percent / 100)` if `net_before_reserve > 0` else `0`; `payout_cents = net_before_reserve - reserve_cents`. If `0 < payout_cents < minimum_payout_cents`: add that pre-clamp `payout_cents` to `reserve_cents`, set `payout_cents = 0` (row is kept).
- Successful stdout: exactly one JSON object with only keys `total_payout_cents`, `total_processing_fee_cents`, `total_dispute_fee_cents`, `total_reserve_cents`, `merchants` (each row: exactly `merchant_id`, `gross_charge_cents`, `refund_cents`, `dispute_cents`, `processing_fee_cents`, `dispute_fee_cents`, `reserve_cents`, `payout_cents`). Totals are the sum of the corresponding final merchant field. No stderr on success.
- Rules loaded at runtime from `data/payout-rules.json` (fields: `processing_fee_percent`, `fixed_fee_cents`, `dispute_fee_cents`, `reserve_percent`, `minimum_payout_cents`) — never hardcoded.

Before declaring PASS, run `npm test` and each of the four additional verification commands in `.devlyn/criteria.generated.md`'s `## Verification` block yourself, and confirm the actual output matches.
