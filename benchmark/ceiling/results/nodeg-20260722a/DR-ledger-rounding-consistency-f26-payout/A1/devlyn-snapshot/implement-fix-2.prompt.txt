<finding>
id: verify-001
severity: HIGH
rule: spec.conflicting-duplicate-precedence
file: bin/cli.js:89 (approximate — the current per-event validation loop)

`.devlyn/criteria.generated.md` Requirement states unconditionally: "The same `id` with differing content is a `conflicting_duplicate` — exit 2, no stdout, stderr is exactly `{"error":"conflicting_duplicate","id":"<id>"}`." The current implementation runs full field validation (unknown `type`, missing `merchant_id`, missing `id`, invalid `amount_cents`) BEFORE checking whether this event's `id` collides with a previously-seen event that has different content. So for input events `[{id:"same",merchant_id:"m",type:"charge",amount_cents:100000},{id:"same",merchant_id:"m",type:"bogus",amount_cents:100000}]`, the second event has both an unknown `type` AND a conflicting-duplicate `id` — the implementation currently reports `{"error":"unknown_type"}` instead of `{"error":"conflicting_duplicate","id":"same"}`.

Reproduction: `printf '%s' '{"events":[{"id":"same","merchant_id":"m","type":"charge","amount_cents":100000},{"id":"same","merchant_id":"m","type":"bogus","amount_cents":100000}]}' > /tmp/verify-repro.json && node bin/cli.js payout --input /tmp/verify-repro.json` currently prints `{"error":"unknown_type"}` (exit 2); must instead print `{"error":"conflicting_duplicate","id":"same"}` (exit 2).

Required precedence: for each event, in array order, check for an `id` collision against a previously-accepted event FIRST (before running this event's own field validation). If the `id` matches a prior event: compare the full raw event objects with deep-equality — identical → skip (idempotent no-op, continue to next event without re-validating or re-applying it); different → `conflicting_duplicate` (exit 2, stderr exactly `{"error":"conflicting_duplicate","id":"<id>"}`, no stdout). Only when there is NO `id` collision does per-field validation (unknown type / missing merchant_id / missing id / invalid amount_cents) run for that event.

Constraint this reorder must preserve: an event whose `id` has never been seen before still goes through full field validation before being accepted into a merchant's totals — this fix changes ONLY the relative order between "check id collision" and "validate this event's own fields", it does not remove any validation.
</finding>
