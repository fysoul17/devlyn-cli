Add a `price-events` command to `bench-cli` so users can run `bench-cli price-events --input <path>` with a JSON file shaped as `{ "events": [{ "id": string, "categoryId": string, "timestamp": number, "basePrice": number }] }`. Pricing rules come from the seeded `data/rule-revisions.json` (an array of `{ "id", "categoryId", "effectiveAt", "discountPct", "minPrice" }` rows, not sorted by category or by time, and not to be modified by the command) — there can be many revisions for the same category over time, and each event must be priced using whichever revision was actually in effect for its category at its own timestamp, not just the newest one.

For each event, find the revision for its category with the greatest `effectiveAt` that is less than or equal to the event's own `timestamp` (an exact match counts). If more than one revision ties on that `effectiveAt`, use the one with the greater `id`. If the category has no revisions at all, reject the event with reason `unknown_category`; if it has revisions but none apply yet at that timestamp, reject with reason `no_effective_rule`. An accepted event's price is the base price discounted by the rule's percentage and rounded, but never below the rule's minimum price.

This command needs to stay fast: downstream billing tooling replays event batches well over a hundred thousand rows against revision sets with many entries per category, and it must still finish within a reasonable process timeout without changing any event's priced or rejected outcome versus a small input processed the same way.

Validate the whole input before pricing anything: `events` must be an array, every event needs a non-empty string `id`/`categoryId`, an integer `timestamp`, and a non-negative integer `basePrice`, and every `id` must be unique. On any violation, exit 2, print exactly one JSON error object to stderr, and print nothing to stdout — use a distinct error for duplicate ids versus other malformed input. On success, print exactly one JSON object with `priced` (id/ruleId/price rows) and `rejected` (id/reason rows), each array keeping the original event order — nothing on stderr.

Update `tests/cli.test.js`: existing tests still pass, plus at least two new tests for `price-events` (one exercising two revisions applying to different events by timestamp, one rejection). No new npm dependencies. Only touch `bin/cli.js` and `tests/cli.test.js`.
