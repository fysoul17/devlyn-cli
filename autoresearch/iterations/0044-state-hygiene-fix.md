## Why-chain (root cause)

1. Why did `phases_ordered` fail? — `build_gate.started_at` (`01:51:44Z`) is chronologically earlier than the *inflated* `implement.completed_at` (`02:15:05Z`), which the checker's monotonic-window logic reads as `build_gate` starting before `implement` (its declared predecessor) finished.
2. Why is `implement.completed_at` inflated relative to `started_at`? — the entry's `completed_at`/`duration_ms`/`round`/`triggered_by`/`verdict` were overwritten to reflect the round-1 (verify-triggered fix-loop) respawn, but `started_at` still carries round-0's original spawn time. `duration_ms` (210000ms) doesn't even match either round's real wall-clock span — direct evidence of a **partial** overwrite, not a full one.
3. Why a partial overwrite? — `references/state-schema.md`'s prior "Write protocol" section (`1. Before each phase spawn: orchestrator writes phases.<name>.{started_at, round, triggered_by}`) was pure prose, hand-executed via the `Edit` tool. Verified empirically against `claude-debug.log` from the flagged run: every `pipeline.state.json` mutation in that run is a raw `Edit`-tool JSON patch — no deterministic script is involved for phase timestamps (the one exception, `verify-merge-findings.py --write-state`, only ever touches `phases.verify.{verdict,sub_verdicts,merged}`).
4. Why does that let `started_at` slip? — nothing computes `duration_ms` from `started_at`; nothing forces `started_at` to refresh together with the completion fields. A model hand-editing the same JSON object across two different phase-agent turns can (and did) touch only the fields it considers "this round's outcome" and leave the rest.

Root invariant violated: **a JSON object written across two different agent turns, with no deterministic script gluing the fields together, has no structural guarantee of internal consistency.** This is the same class of failure this repo has hit repeatedly on prompt-only state contracts (iter-0006 through iter-0014: "prompt-level contracts die, mechanisms survive").

A second contributing finding surfaced during Codex cross-check (not in the original brief): the phase **worker's own prompt** (`references/phases/{plan,implement,build-gate,cleanup}.md`) told the worker itself to "update `pipeline.state.json:phases.X.{verdict, completed_at, duration_ms}`" in its `<output>` section — i.e. the ambiguity wasn't only "orchestrator hand-edits sloppily," it was "the contract doesn't even clearly say the orchestrator (not the worker) owns this write." Fixing only the orchestrator-facing docs (SKILL.md / state-schema.md) would have left this bypass open.

## Mechanism vs. prose — decision + criterion

**Decision**: extract phase-lifecycle writes into a new deterministic script, `config/skills/_shared/state-phase-write.py`, with two subcommands:

- `spawn --round N [--triggered-by build_gate|verify] [--pre-sha] [--engine] [--model]` — always sets `started_at = now()` fresh and **nulls** `completed_at`/`duration_ms`/`verdict`/`artifacts`/`sub_verdicts`. A respawn can never inherit a prior round's stale timing or verdict — if it crashes before `complete` runs, the fields are visibly null (loudly caught by the existing `phases_ordered`/null-verdict check) rather than silently carrying forward plausible-but-wrong data. Merges into the existing `phases.<name>` dict rather than replacing it, so fields the script doesn't own (`phases.implement.exec`, the phase-gated large-run progress tracker) survive untouched.
- `complete --verdict V [--findings-file] [--log-file]` — requires the phase was already spawned (errors loudly otherwise — no silent invention). Derives `duration_ms` from the phase's own recorded `started_at`, so it structurally cannot drift from `completed_at` again. `--verdict` is required for every phase except `verify`: `verify-merge-findings.py --write-state` is VERIFY's sole verdict writer, so `complete verify` runs with no `--verdict` (an explicit one is rejected outright) and preserves whatever is already on disk.

**Rejected alternative**: `phases.<name>.rounds[]` — an array recording every round's own sub-object, keeping the top-level phase fields as a "latest round" summary.

**Named criterion for rejecting it**: no current consumer (terminal-verdict precedence, the final-report per-phase table, the compliance probe's `phases_ordered` check, `verify-merge-findings.py`) reads anything about a phase's *earlier* rounds — every reader only ever looks at the latest entry. The observed defect isn't "we lost round-0's history" (that's already true today by design, and nothing has ever needed it) — it's "the round-1 overwrite was partial/inconsistent." A deterministic writer fixes exactly that with a smaller diff (no new field, no schema-shape change, no downstream-reader change) than `rounds[]` would require, and it keeps parity with existing precedent (VERIFY's `merged`/`sub_verdicts` are already deterministic-script-owned; this extends the same ownership model to timestamps, for every phase, instead of introducing a second pattern). `rounds[]` would have been overengineering per CLAUDE.md's "new abstractions require an observed failure mode they prevent" — none of the observed failure requires round history to be queryable.

## A bug in the fix itself, caught by its own self-test

Running `state-phase-write.py --self-test` repeatedly (not just once) surfaced a real, if minor, flakiness: `now_iso()` formats a timestamp with millisecond precision by *truncating* microseconds (`dt.microsecond // 1000`), but the original `do_complete()` computed `duration_ms` from the full-microsecond-precision `now` object before truncation — so the stored `duration_ms` could be off by up to 1ms from what re-parsing the stored `completed_at` string would compute. This is the exact class of bug the whole fix exists to eliminate (`duration_ms` drifting from `completed_at`/`started_at`), just at a much smaller scale than iter-0042's ~200-second mismatch. Fixed by capturing a millisecond-truncated `now` once (`now_ms()`) and using that single value for both the stored string and the duration calculation, so they're structurally identical instead of independently rounded. Verified with 20 consecutive self-test runs (previously flaky roughly 1-in-a-few).

## Codex convergence (3 rounds, `model_reasoning_effort=high`, read-only)

**Round 1** — confirmed root cause and the `rounds[]` rejection; confirmed the verify-null symptom is unreproduced. Delta from my initial draft: (a) `complete --verdict` must be required for every phase except VERIFY, not optional-everywhere; (b) `spawn` must merge, not replace, or it silently clobbers `phases.implement.exec`; (c) timestamps need normalized fractional-second width for the compliance checker's plain string comparison; (d) reject `BLOCKED:<reason>`-style verdicts (already handled structurally by the closed enum); (e) — the important one — the **phase-canonical-body files** (`plan.md`, `implement.md`, `build-gate.md`, `cleanup.md`, `verify.md`) are the workers' own prompts and directly instructed the worker to write `pipeline.state.json` itself; fixing only `SKILL.md`/`state-schema.md` would leave that path live.

**Round 2** — presented the refined script + exact doc-edit plan (single clarifying sentence in SKILL.md's `## State management` rather than touching all 7 "State write:" one-liners, plus the 5 phase-file edits). Confirmed: single sentence is sufficient; phase-file edits are correct and the VERIFY split (verdict/sub_verdicts/merged via `verify-merge-findings.py`, timing/artifacts via the orchestrator) is right; `rounds.global` (the separate top-level fix-loop counter) correctly stays untouched — no observed root cause ties it to this defect. One final guard added: `complete verify` must *reject* an explicit `--verdict` rather than silently accept-and-ignore it, since `phases.verify.verdict` is owned exclusively by `verify-merge-findings.py --write-state`.

**Round 3** — live verification surfaced a real problem: two live `/devlyn:resolve` compliance-cell re-runs (`claude-small`, `claude-medium`) both PASSed, but neither naturally hit a fix-loop this time (all phases stayed at round 0) — an "easy pass" that doesn't exercise the fix, exactly the trap the team lead's brief warned against. I built a controlled reconstruction of the exact iter-0042 shape using real `state-phase-write.py` CLI invocations (not fabricated JSON): `plan → implement(r0) → build_gate(r0) → cleanup(r0) → verify(r0, NEEDS_WORK) → implement(r1, respawn) → verify(r1, PASS) → final_report`. Running `check_phases_ordered` against this **still FAILed with `out_of_order_phase: "build_gate"` even with my fix applied** — a mechanism-independent false positive. Root cause: the checker chains `prev_end` strictly through `DECLARED_PHASE_ORDER`, comparing `build_gate.started_at` against `implement.completed_at`. Once VERIFY's fix loop respawns IMPLEMENT to round 1 without re-running BUILD_GATE/CLEANUP (confirmed as the harness's actual, intentional behavior — `SKILL.md` PHASE 5's VERIFY branch only respawns IMPLEMENT, and real iter-0042 production data shows `build_gate`/`cleanup` staying at `round: 0`), `implement.completed_at` is *legitimately* later than `build_gate.completed_at` — refreshing `started_at` (my fix) does nothing for this specific comparison, since it never touches `completed_at`.

Presented this to Codex with the concrete controlled-test evidence. Confirmed: fix the checker (option A), not the schema — reintroducing round history (`rounds[]` or a `first_started_at` field) was explicitly rejected as unnecessary; a `rounds[]` array wouldn't even help since the existing checker only reads the flat/latest fields, so the checker itself has to change regardless of schema shape. Codex narrowed my initial over-broad proposal ("skip whenever any upstream phase's round exceeds a downstream phase's") to the precise condition actually justified by evidence: skip the `build_gate` comparison *only* when `implement.triggered_by == "verify"` and `implement.round > build_gate.round` — the exact fingerprint of a verify-triggered, build-gate-not-rerun respawn. A BUILD_GATE-triggered fix loop (where BUILD_GATE itself re-runs and its round keeps pace with IMPLEMENT's) does not match this condition and is checked normally.

All three rounds converged with no unresolved disagreement.
