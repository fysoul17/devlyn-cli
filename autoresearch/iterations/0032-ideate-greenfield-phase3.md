# iter-0032 — greenfield `/devlyn:ideate`

## Why this iter exists (PRINCIPLES.md pre-flight 0)

Phase 3 of the 2-skill redesign. iter-0029 shipped the kernel (Phase 1). iter-0031 shipped greenfield `/devlyn:resolve` (Phase 2). This iter ships greenfield `/devlyn:ideate` — completing the 2-skill user-facing pair.

Per NORTH-STAR.md "Product surface": `/devlyn:ideate` is OPTIONAL — `/devlyn:resolve` is standalone-capable via free-form mode and `--spec` mode. The reason `/devlyn:ideate` exists is to give context-engineering-naive users a surface that AI drives via questions until a verifiable spec exists.

Current `/devlyn:ideate` (334 lines + 4 reference files = 664 lines) carries Vision/Roadmap multi-doc architecture from earlier iterations. The locked design is tighter: single-feature spec by default, `--project` flag for multi-feature.

## Mission 1 service (PRINCIPLES.md #7)

Single-task L1 surface. No Mission 2 substrate. Hard NOs untouched.

## Hypothesis

**Structural hypothesis**: A greenfield `/devlyn:ideate` SKILL.md focused on single-spec extraction (with `--project` for multi-feature, `--from-spec` for external normalization, `--quick` for assume-and-confirm) will:
- Produce specs that `/devlyn:resolve --spec <path>` consumes without translation.
- Use the iter-0029 schema (`_shared/expected.schema.json`) as the contract for `spec.expected.json` output.
- Honor each model's official prompt-engineering guide via the iter-0029 adapter system.
- Drop Vision/Roadmap concepts that produce context pollution for non-engineers.

**Falsifiable predictions** (acceptance gate):
1. `bash scripts/lint-skills.sh` exit 0.
2. SKILL.md frontmatter parseable (name + description).
3. SKILL.md ≤ 250 lines.
4. Zero NEVER/MUST/CRITICAL imperatives at line start.
5. Zero iter-history annotations.
6. Mirror parity for all new files.
7. A test spec generated by following the SKILL.md contract validates with `python3 .claude/skills/_shared/spec-verify-check.py --check <test-spec.md>` exit 0.
